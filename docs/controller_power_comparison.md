# Controller power comparison (ω = 1–4 rad/s)

This file is generated/updated by:

```bash
python3 scripts/controller_power_sweep.py
```

The script runs the full headless controller-period sweep (H = 0.028 m regular waves), writes:
- `output/controller_power_sweep.csv` — with `P_mean_W`, `regime` (`absorbing`/`reactive_limited`), `note`, and `peak_abs_inst_power_W` columns
- `output/controller_power_sweep.png` — reactive-limited CC points shown as hollow grey markers at ≈0
- `docs/controller_power_comparison.md` (this file, with results table + interpretation)

Run the command above in an environment where `build/demo_vgoswec` is available.

---

## Setup

- Regular waves with fixed height H = 0.028 m (A = H/2 = 0.014 m) for all periods/controllers.
- Period sweep: 6.00, 4.49, 3.42, 3.00, 2.50, 2.00, 1.57 s (ω ≈ 1.05–4.00 rad/s).
- Controllers: passive, opt_passive, cc, and exc_ff_pid (if run was stable).
- Hydro coefficients from de-normalized H5 accessors (BEMIO convention; rho_eff ≈ 1002.7 kg/m³).
- Steady-state averaging: discard first 10 wave periods; average over an **integer number of
  complete wave cycles** up to 50 total periods. Integer-cycle truncation removes partial-cycle
  bias in reactive power, which is the main cause of spurious sign flips for reactive-limited CC.
- Sign convention: absorbed power is positive; computed from `power_w` with a global sign
  check against `−τ·ω`.
- CC reactive-limited threshold: **B_R_FLOOR = 1e-4 N·m·s/rad** (mirrored in
  `scripts/controller_power_sweep.py` and `src/demo_vgoswec.cpp`).

## Key findings

### CC at the radiation-damping notch (T ≈ 3.42 s, ω₀ ≈ 1.84 rad/s)

The VGOSWEC-45 pitch radiation damping **B₅₅(ω₀) ≈ 2.19×10⁻⁶ N·m·s/rad** at ω₀ = 1.84 rad/s —
more than 4 orders of magnitude below the reactive-limited threshold of 1×10⁻⁴.

With B_r = B₅₅(ω₀) ≈ 0, the CC control law **τ = −K_r·θ − B_r·ω degenerates to a pure reactive
spring τ ≈ −K_r·θ**. A pure spring does exactly zero net work over a complete cycle. The residual
(typically small negative) value seen in the raw P_mean_W column is **numerical noise** about true
zero, amplified by the very large flap motion at resonance (peak |θ| ≈ 0.847 rad ≈ 49°). This is
**not a bug** in the control law and **not a mistuning** — the CC gains are correctly computed at
the incident wave frequency for each sweep point (confirmed: `omega0 = 1.84 rad/s` prints when
running at `--wave-period 3.42`).

These points are:
- Marked `regime = reactive_limited` in `output/controller_power_sweep.csv` (raw P_mean_W preserved).
- Reported as **`~0 (reactive-limited)`** in the stdout table and results markdown.
- Shown as **hollow grey markers at ≈0** in the plot with an annotation.
- Flagged by a `⚠ CC degenerate` warning printed by `demo_vgoswec` at runtime.

### CC away from the notch

Away from the radiation-damping notch (e.g. T = 1.57 s, ω ≈ 4.0 rad/s), CC correctly absorbs a
small positive net power. These points are marked `regime = absorbing`.

### Passive / opt_passive across the band

Both passive and opt_passive controllers absorb positive power across the entire swept band
(τ·ω < 0 fraction ≈ 1.0 — they always oppose motion, never inject energy). Passive controllers
typically deliver the highest absolute absorbed power in this low-amplitude, low-radiation-damping
regime, while CC collapses to a reactive trap near the resonance.

### exc_ff_pid

The exc_ff_pid controller is included in the sweep if its probe run is stable. See the regenerated
table for per-period results.

## CSV schema additions

The `output/controller_power_sweep.csv` (not tracked in git; regenerated by the sweep script)
adds the following columns relative to the previous schema:

| New column | Type | Description |
|---|---|---|
| `peak_abs_inst_power_W` | float | Max \|power_w\| in steady-state window; diagnostic for reactive swings |
| `regime` | string | `absorbing` or `reactive_limited` |
| `note` | string | Human-readable explanation for reactive-limited rows |

The simulation output schema (`time_s`, `flap_pitch_rad`, `flap_pitch_vel_rads`, `pto_torque_nm`,
`exc_torque_nm`, `power_w`) is **unchanged**.
